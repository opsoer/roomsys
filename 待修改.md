# 项目待修改清单

基于对代码的全面扫描和业务理解，整理出以下待修改问题，按优先级分类。

---

## 一、安全问题 (P0 - 立即修复)

### 1.3 CORS 配置过于宽松 ✅
- **位置**: `server/main.go:96-124`
- **问题**: `Access-Control-Allow-Origin: *` 允许任意来源跨域请求
- **修复方案**:
  - 配置白名单域名，仅允许已知前端域名
  - 生产环境不应使用通配符 `*`

### 1.4 Token 存储在 localStorage ✅
- **位置**: `web/src/views/Login.vue`, `web/src/api/index.js`
- **问题**: localStorage 可被同源域的任意 JS 访问，若存在 XSS 漏洞，攻击者可窃取 token
- **修复方案**:
  - 使用 httpOnly + Secure + SameSite cookie 存储 JWT
  - 或至少封装到 Pinia store 中统一管理

### 1.5 路由守卫仅依赖 localStorage 中的 role ✅
- **位置**: `web/src/router/index.js:73-105`
- **问题**: 用户可在浏览器 DevTools 中将 `role` 改为 `super_admin` 绕过前端路由守卫
- **修复方案**:
  - 路由守卫配合服务端鉴权（调用 `/auth/me` 接口验证 token 有效性）
  - 不仅依赖 localStorage 的静态值

### 1.6 Building Scope 身份验证缺陷 ✅
- **位置**: `server/middleware/auth.go:124-169`
- **问题**: `BuildingScopeMiddleware` 仅检查 `building_id` 是否存在，不验证用户是否真的属于该 building
- **修复方案**:
  - 验证 JWT 中的 `building_id` 与数据库中用户的 `BuildingID` 一致
  - 或在 AuthMiddleware 中从数据库加载最新用户信息

## 二、性能问题 (P1 - 尽快修复)

### 2.1 N+1 查询问题 ✅
- **位置**: `server/handlers/buildings.go`
- **问题**: 每个 building 执行 5 个额外查询，100 个 building 会执行 501 个 SQL
- **修复方案**:
  - 使用 `Preload` 批量加载
  - 使用子查询一次性统计所有 building 的房间数据
  - 使用 Redis 缓存统计数据

### 2.2 列表无分页 ✅
- **位置**: 多个前端页面
- **问题**: 一次性加载全部数据（房间、账单、任务等），数据量大时页面卡顿
- **修复方案**:
  - 引入分页或无限滚动
  - API 层支持 `page`/`page_size` 参数

### 2.4 DATE_FORMAT 导致索引失效 ✅
- **位置**: `server/handlers/bills.go:351-364`
- **问题**: 对列使用函数导致 MySQL 无法使用索引
- **修复方案**:
  - 改为范围查询：`WHERE bill_date >= ? AND bill_date < ?`

### 2.5 图片无懒加载 ✅
- **位置**: `BuildingList.vue`, `BuildingPublic.vue`, `RoomList.vue`
- **问题**: 所有图片立即加载，无 `loading="lazy"` 属性
- **修复方案**:
  - 为 `<img>` 添加 `loading="lazy"` 属性

## 三、代码质量问题 (P1-P2)

### 3.1 事务 defer + recover 模式不可靠 ✅
- **位置**: 多个 Handler 文件
- **问题**: defer 中的 `tx.Rollback()` 在正常 return 时可能被调用，导致已提交事务被回滚
- **修复方案**:
  - 统一使用 `defer func() { if tx.Error != nil { tx.Rollback() } }()` 模式
  - 或封装事务工具函数

### 3.2 大量重复函数 ✅
- **位置**: 前端多个组件
- **问题**: `mediaUrl()`, `goToDashboard()`, `logout()`, `statusLabel()` 等函数在多处重复
- **修复方案**:
  - 提取到 `composables/useUtils.js` 中

### 3.3 无状态管理库 ✅
- **位置**: 整个前端项目
- **问题**: 用户状态在至少 8 个文件中通过 `localStorage.getItem()` 直接读取
- **修复方案**:
  - 引入 Pinia store 统一管理用户状态

### 3.4 组件过大 ✅
- **位置**: `RoomDetail.vue` (779行), `Bills.vue` (739行), `AdminBuildings.vue` (528行)
- **问题**: 单文件包含多个 Dialog 和完整表单，难以维护
- **修复方案**:
  - RoomDetail.vue → `components/room/` (RoomHero, RoomMedia, RoomSidebar, RoomDialogs)
  - Bills.vue → `components/bill/` (BillList, BillStats, BillTrend, BillPredict, BillDialog)
  - AdminBuildings.vue → `components/admin/` (AdminBuildingList, AdminBuildingDialogs)

### 3.5 错误处理不一致 ✅
- **位置**: 前端多个组件
- **问题**: 混合使用 `ElMessage.error()`, `showToast()`, 空 `catch {}`
- **修复方案**:
  - 统一错误处理策略
  - 添加全局 `app.config.errorHandler`

### 3.6 重复代码严重 ✅
- **位置**: 后端多个 Handler
- **问题**: 大量重复的 `c.Get("building_id")` + 类型断言模式
- **修复方案**:
  - 封装 `GetBuildingID(c *gin.Context) (uint, error)` 工具函数

## 四、业务逻辑问题 (P2)

### 4.2 房间状态更新的竞态条件 ✅
- **位置**: `server/handlers/rooms.go:366`
- **问题**: 并发操作可能导致两人同时看到房间状态为 `vacant`，都执行 `rented` 操作
- **修复方案**:
  - 使用悲观锁（`SELECT ... FOR UPDATE`）

### 4.5 公开 API 泄露房间合同信息 ✅
- **位置**: `server/handlers/rooms.go:83-98`
- **问题**: 已登录用户可通过构造请求查看其他 building 的合同详情
- **修复方案**:
  - 添加 `building_id` 的过滤验证

### 4.6 删除建筑时未删除 Tenant ✅
- **位置**: `server/handlers/buildings.go:281-287`
- **问题**: 级联删除了多个表，但没有删除关联的 Tenant 记录
- **修复方案**:
  - 添加 Tenant 清理逻辑或使用软删除

## 五、架构设计问题 (P2-P3)

### 5.1 缺少 Service 层 ✅
- **位置**: 整个后端
- **问题**: 所有业务逻辑都在 Handler 中直接调用 `h.DB`，导致业务逻辑无法被测试
- **修复方案**:
  - 引入 Handler → Service → DB 分层架构
  - 创建 services 包：BuildingService, RoomService, BillService, AuthService, DividendService, TaskService, SettingsService, RecruitService, MediaService
  - 更新 routes.go 注入 services
  - 更新所有 handlers 使用 services

### 5.3 缺少 Composable 层 ✅
- **位置**: 整个前端
- **问题**: 所有跨组件逻辑通过复制粘贴实现，零个 composable
- **修复方案**:
  - 创建 `composables/useUtils.js` 封装可复用逻辑

### 5.4 API 层设计不一致 ✅
- **位置**: `web/src/api/index.js`
- **问题**: 约 6 个 API 调用绕过了命名函数，直接使用 `api` 实例
- **修复方案**:
  - 所有 API 调用必须通过命名函数
  - 已添加缺失的命名函数并更新所有组件

### 5.5 无环境变量配置 ✅
- **位置**: `vite.config.js`, `web/src/api/index.js`
- **问题**: API 基础 URL 和超时时间硬编码
- **修复方案**:
  - 使用 Vite 的 `import.meta.env` 管理环境变量

### 5.6 Vant + Element Plus 双框架包体积 ✅
- **位置**: `web/src/main.js`
- **问题**: 同时引入两个 UI 框架，总体 JS bundle 较大
- **修复方案**:
  - 使用 `unplugin-vue-components` + `unplugin-auto-import` 按需导入
  - Tree-shaking 移除未使用组件

## 七、其他问题

### 7.1 CSV 导出缺少 UTF-8 BOM ✅
- **位置**: `server/handlers/bills.go:267`
- **问题**: 中文 CSV 文件 Excel 打开会乱码
- **修复方案**:
  - 添加 UTF-8 BOM 头

### 7.2 登录频率限制内存泄漏 ✅
- **位置**: `server/handlers/auth.go:27-45`
- **问题**: `sync.Map` 无过期清理机制，内存会持续增长
- **修复方案**:

### 7.3 Bill No 序列号取模溢出 ✅
- **位置**: `server/utils/time.go:44`
- **问题**: 序列号回绕时可能产生重复的 BillNo
- **修复方案**:
  - 改为 5 位序列号 `% 100000`

### 7.4 缺少测试
- **位置**: 整个项目
- **问题**: 无任何测试文件，无测试框架配置
- **修复方案**:
  - 添加 vitest 测试框架
  - 为工具函数添加单元测试

### 7.5 无 CSRF 防护机制 ✅
- **位置**: 整个系统
- **问题**: 所有写操作无 CSRF Token 验证
- **修复方案**:
